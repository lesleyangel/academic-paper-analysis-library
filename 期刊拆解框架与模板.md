# 期刊拆解框架与模板

这份文档用于把多篇单篇论文拆解结果汇总成一个期刊画像。它继承 `论文拆解框架与模板.md` 的核心思路：不是总结期刊“发什么”，而是还原这个期刊如何判断一篇论文值得发表，偏好什么问题、什么证据、什么理论/技术组合、什么写作结构、什么作者与团队形态。

核心单位仍然是单篇论文。每篇论文是一个样本点，期刊画像是由样本点抽象出来的稳定偏好。

## 一、核心原则

拆期刊要同时看三层：

1. **论文层**：每篇被录用论文的选题、方法、证据、结构、文笔和审稿风险。
2. **聚合层**：多篇论文之间反复出现的研究对象、问题来源、技术路径、理论框架、标题句式、引用组织和图表套路。
3. **门槛层**：这个期刊为什么愿意接收这些论文，不愿意接收什么类型的论文。

不要只问“这个期刊研究什么方向”，而要追问：

- 它喜欢什么样的问题定义？
- 它认为哪些 gap 值得发表？
- 它偏好理论贡献、方法贡献、工程贡献、数据贡献，还是综合贡献？
- 它接受多强的创新，要求多强的证据？
- 它喜欢什么样的标题、摘要、Introduction 和 Discussion？
- 它的常用词是不是体现了一种稳定的学术气质？
- 作者通常是什么画像：单一团队、跨学科合作、资深团队、年轻团队、工程机构、实验室联盟？
- 一篇论文为什么能被它录用？如果换到别的期刊，会不会成立？

## 二、最终产物

一个完整的期刊拆解，建议输出 8 个产物：

1. **期刊一句话画像**：这个期刊本质上奖励什么。
2. **研究方向地图**：方向、对象、场景、方法、理论、证据的分布。
3. **录用论文类型谱系**：理论型、方法型、实验型、模型型、工程应用型、综述型等。
4. **写作风格画像**：标题、摘要、引言、结构、句式、语气、图表、引用。
5. **常用词与常用句式库**：按功能分类，而不是只做词频。
6. **作者与团队画像**：谁在这个期刊上发，靠什么资源和积累发。
7. **录用机制解释**：为什么这些论文能过审。
8. **投稿反推清单**：如果我想投这个期刊，论文必须长成什么样。

## 三、样本选择

样本边界决定期刊画像的可信度。

### 1. 样本范围

可选策略：

- **近期快照**：近 1 年或近 2 年全部论文，适合判断当前偏好。
- **主题样本**：某一关键词、专栏、方向下的论文，适合判断子领域偏好。
- **高被引样本**：高影响论文，适合判断期刊的理想贡献形态。
- **目标相似样本**：与你的研究问题、方法或对象相似的论文，适合反推投稿策略。
- **拒稿对照样本**：如果有 rejected/转投论文，可辅助判断边界。

### 2. 样本量建议

```text
最小样本：10 篇
标准样本：20-40 篇
深度样本：50 篇以上
主题专项：同一方向 8-15 篇即可形成初步画像
```

### 3. 样本记录表

| 字段 | 记录内容 |
| --- | --- |
| Paper ID | 文件名或编号 |
| 标题 | 原标题 |
| 年份 | 发表年份 |
| 论文类型 | 理论/方法/实验/数值/综述/案例/数据/观点 |
| 研究对象 | 材料、系统、结构、算法、场景、群体等 |
| 问题来源 | 理论空白、工程瓶颈、方法不足、数据缺口等 |
| 主要方法 | 实验、模型、仿真、推导、统计、机器学习等 |
| 主要证据 | 图表、公式、实验、案例、数据集、对比、消融等 |
| 贡献类型 | 理论/方法/证据/工程/整合/表达 |
| 作者画像 | 机构、国家/地区、团队规模、合作类型 |
| 录用判断 | 为什么这篇适合该期刊 |

## 四、期刊身份层

先回答这个期刊“在学术生态里扮演什么角色”。

可填写：

```text
期刊名称：
学科位置：
主要读者：
核心问题域：
常见研究对象：
常见方法体系：
常见证据类型：
偏理论 / 偏方法 / 偏工程 / 偏综合：
是否偏跨学科：
是否偏机制解释：
是否偏可复现系统：
是否偏真实案例：
```

关键判断：

- 它是奖励“新机制”，还是奖励“可用方法”？
- 它是奖励“严谨理论”，还是奖励“工程信息组织能力”？
- 它喜欢大而完整的框架，还是小而锋利的发现？
- 它对实际应用的要求是“真实部署”，还是“案例验证即可”？
- 它是否容忍单案例？如果容忍，单案例必须提供什么额外价值？

## 六、选题偏好

期刊的选题偏好，核心看它承认什么样的“值得研究”。

### 1. 问题来源偏好

| 问题来源 | 观察问题 | 期刊偏好判断 |
| --- | --- | --- |
| 理论空白 | 是否反复奖励机制、定理、模型边界 | 强/中/弱 |
| 方法不足 | 是否喜欢新算法、新流程、新测量 | 强/中/弱 |
| 工程瓶颈 | 是否要求实际系统、真实约束、部署问题 | 强/中/弱 |
| 数据缺口 | 是否接受新数据集、新观测、新 benchmark | 强/中/弱 |
| 表示缺口 | 是否喜欢信息模型、知识组织、语义结构 | 强/中/弱 |
| 整合缺口 | 是否喜欢 taxonomy、framework、review、unified model | 强/中/弱 |

### 2. Gap 类型偏好

```text
最常见 gap 类型：
最容易被接受的 gap：
经常被包装但风险较高的 gap：
需要强证据才能成立的 gap：
期刊不太买账的 gap：
```

判断标准：

- gap 是对象缺失、方法缺失、机制缺失、数据缺失，还是理论缺失？
- gap 是否从文献版图自然推出？
- gap 是否刚好对应期刊关注的读者痛点？
- gap 是否能被本文证据闭合？

这一层回答：期刊喜欢论文“靠什么东西成立”。

### 1. 技术偏好

```text
是否喜欢端到端系统：
是否喜欢可解释模型：
是否喜欢物理约束/理论约束：
是否喜欢 AI/数据驱动方法：
是否要求工程可部署性：
```

### 2. 理论偏好

```text
常见理论基底：
理论使用方式：继承 / 修正 / 扩展 / 替代 / 整合
是否偏好机制解释：
是否偏好形式化定义：
是否偏好数学推导：
是否偏好概念框架：
是否允许弱理论强案例：
是否允许强理论弱案例：
```

### 3. 技术-理论组合

常见组合可按下面方式归类：

| 类型 | 典型结构 | 期刊接受条件 |
| --- | --- | --- |
| 理论 + 数值验证 | 推导 -> 模拟 -> 参数/边界分析 | 推导清楚，验证覆盖关键边界 |
| 模型 + 实验验证 | 模型 -> 实验 -> 对比解释 | 实验能支撑模型假设 |
| 数据驱动 + 物理解释 | ML -> 解释 -> 工程意义 | 不只追精度，要解释机制或任务价值 |
| 框架 + 案例 | representation/workflow -> case | 框架能迁移，案例能闭合 |
| 综述 + taxonomy | 文献筛选 -> 分类 -> 挑战 | 分类维度有新信息，而非罗列 |

## 八、创新偏好

统计每篇论文的真实创新类型，而不是只看作者声明。

| 创新类型 | 期刊是否偏好 | 录用时通常需要什么证据 |
| --- | --- | --- |
| 问题创新 | 强/中/弱 | 清楚的问题定义和领域必要性 |
| 理论创新 | 强/中/弱 | 推导、机制、边界、与经典理论关系 |
| 方法创新 | 强/中/弱 | baseline、消融、误差、复杂度、复现性 |
| 数据创新 | 强/中/弱 | 数据质量、覆盖范围、代表性、可用性 |
| 工程创新 | 强/中/弱 | 真实场景、约束、任务输出、决策价值 |
| 整合创新 | 强/中/弱 | taxonomy、framework、跨文献结构 |
| 表达创新 | 强/中/弱 | 新指标、新图式、新结构是否降低理解成本 |

可填写：

```text
该期刊最常奖励的创新组合：
单一创新是否足够：
弱创新靠什么补强：
最危险的伪创新：
高频贡献表达：
低频但高价值贡献表达：
```

### 2. Introduction 结构

记录常见推进链：

```text

工程需求 -> 方法不足 -> 本文框架 -> 案例验证
现象观察 -> 机制不清 -> 理论/模型 -> 验证
```

期刊偏好判断：

- 文献版图是长还是短？
- gap 是集中一段提出，还是穿插在文献综述中？
- contribution 是 bullet 还是自然段？
- 是否需要 organization paragraph？

```text

是否强调可复现细节：

```


## 十八、最小可用版本

如果时间很少，只回答 12 个问题：

```text
1. 这个期刊一句话奖励什么？
2. 它最常发哪几类论文？
3. 它最喜欢什么研究对象和工程/理论场景？
4. 它最认可什么 gap？
5. 它最常接受什么创新类型？
6. 它要求什么证据强度？
7. 它的 Introduction 通常怎么制造问题？
8. 它的标题通常怎么写？
9. 它的常用词和语气有什么稳定偏好？
10. 它如何使用引用制造可信度？
11. 哪些作者/团队最容易在这里发表，靠什么？
12. 如果我要投它，最需要补齐哪三件事？
```

## 十九、一句话总结

拆期刊，不是把多篇论文做成主题列表，而是从每篇论文里抽取“被该期刊认可的学术生产方式”：什么问题值得写，什么创新算数，什么证据够强，什么语言可信，什么作者资源能支撑，以及一篇论文为什么在这个期刊语境里显得“应该被录用”。
